# Definition of Done：防误报验收纪律

本文档用于约束项目修复、上线前治理和验收报告，目标是避免“代码或测试显示通过，但真实后台体验仍失败”的误报。

## 功能完成定义

一个功能或修复只有在对应链路完整闭环后，才能报告为“已修复”：

1. 后端逻辑完成。
2. 页面入口完成。
3. 表单提交完成。
4. 数据保存完成。
5. 页面回显完成。
6. 权限控制完成。
7. 错误提示完成。
8. 人工体验确认完成。

如果问题属于 UI、视觉或交互体验，但当前无法进行真实浏览器确认，只能报告为“代码层已调整，待人工确认”，不能报告为“已修复”。

## 不允许作为完成证明的内容

以下内容只能作为辅助信息，不能单独证明修复完成：

1. 只说 `pytest passed`。
2. 只说 HTML 能搜到文案。
3. 只说路由存在。
4. 只说临时库通过。
5. 只说没有 404/500。
6. 只说代码已改。

自动化测试、HTML 搜索、路由探测和临时数据库验证都不能替代真实页面体验。

## 必须作为完成证明的内容

修复报告必须尽量提供真实链路证据：

1. 真实页面路径。
2. 真实模块名称。
3. 真实按钮文案。
4. 真实 `form action`。
5. 真实字段名和值。
6. 真实 POST 返回。
7. 真实 DB 结果。
8. 真实页面回显。
9. 如果是 UI 问题，必须人工确认，或明确标记“仍需人工确认”。

当真实页面人工验收不通过时，即使自动化测试通过，也不能提交。

## 验收分级

### A 类：自动化可确认

以下问题可以主要通过自动化、接口、数据库或只读验收确认：

1. 404/500。
2. 权限。
3. POST。
4. DB 保存。
5. 导出。
6. 数据范围。

### B 类：必须人工确认

以下问题必须以人工真实浏览器体验为最终准绳：

1. 视觉错位。
2. 空白。
3. 卡顿。
4. 弹窗。
5. 下拉。
6. 宽表滑动。
7. 移动端。
8. 展开/收起。

B 类问题可以补充截图、HTML 结构、CSS 检查和脚本验证，但这些都不能替代人工确认。

## 报告结论规范

报告最终结论只能使用以下语义：

1. 已修复：必须有真实链路证据，且人工体验或等效真实链路验收通过。
2. 代码层已调整，待人工确认：用于 UI、视觉、交互问题，当前没有人工确认结果。
3. 未完成：无法复现、无法定位、验证失败或还有必要链路未闭环。
4. 不通过：人工确认失败，或真实页面仍存在用户报告的问题。

不得把“自动化测试通过”写成 UI/交互问题的最终修复证明。

## 修复类任务回复格式

每次修复类任务回复必须包含：

1. 是否复现。
2. 复现路径。
3. 根因。
4. 修复文件。
5. 自动化验证结果。
6. 真实页面链路验证结果。
7. 是否需要人工确认。
8. 是否可以提交。
9. 如果人工未确认，不得建议提交。

## 破坏性操作纪律

所有破坏性操作必须满足：

1. 先备份。
2. 先 dry-run。
3. 必须有明确 confirm。
4. 不得默认执行删除。
5. 不得删除 `data/tickets.db`、`.env`、`uploads/`、`backups/` 或用户未明确授权删除的运行数据。

清理、重置、删除类脚本的报告必须写清楚备份目录、dry-run 结果、执行参数和真实影响范围。
